# Publishing an Obsidian Plugin

Guide for releasing and submitting an Obsidian plugin to the community registry.

## Workflow

1. Complete the pre-release checklist below
2. Update `manifest.json` version and `versions.json`
3. Run `npm run build` for production output
4. Create a GitHub release with required assets
5. Submit a PR to the community registry
6. Address reviewer feedback on the existing PR

## Pre-Release Checklist

Before submitting, verify:

**Required files:**
- [ ] `README.md` describes purpose, features, and usage
- [ ] `LICENSE` file exists (MIT recommended)
- [ ] `manifest.json` has correct `id`, `name`, `version`, `minAppVersion`
- [ ] Plugin ID is kebab-case and unique
- [ ] Class names are descriptive (not `MyPlugin` or `SampleSettingTab`)

**Code quality (common rejection reasons):**
- [ ] No excessive `console.log` calls (only errors in production)
- [ ] No `innerHTML`/`outerHTML`/`insertAdjacentHTML` with user input
- [ ] Uses `this.app` instead of global `app` object
- [ ] All events registered with `registerEvent()` or `registerDomEvent()`
- [ ] No default hotkeys set on commands
- [ ] Uses `getActiveViewOfType()` instead of `workspace.activeLeaf`
- [ ] File operations use Vault API, not Adapter API
- [ ] Uses `normalizePath()` for user-provided paths
- [ ] Frontmatter edits use `FileManager.processFrontMatter()`
- [ ] Active note edits use Editor API, not `Vault.modify()`
- [ ] Settings UI follows Obsidian conventions (sentence case, `setHeading()`)
- [ ] If mobile-compatible: no Node/Electron APIs, no regex lookbehind

## versions.json

Maps plugin versions to minimum Obsidian versions:

```json
{
  "1.0.0": "0.15.0",
  "1.1.0": "1.0.0"
}
```

Update this when you change `minAppVersion` in manifest.json. The `version-bump.mjs` script (generated by the `project-setup` skill) handles this automatically via `npm version`.

## Creating a GitHub Release

1. Update `manifest.json` version (semantic versioning)
2. Update `versions.json` if `minAppVersion` changed
3. Run `npm run build` to produce production `main.js`
4. Create a GitHub release:
   - Tag must match the version in `manifest.json` (e.g., `1.0.0`)
   - Attach as binary assets:
     - `main.js` (required)
     - `manifest.json` (required)
     - `styles.css` (if your plugin has styles)

```bash
# Build for production
npm run build

# Create release with gh CLI
gh release create 1.0.0 \
  main.js \
  manifest.json \
  styles.css \
  --title "1.0.0" \
  --notes "Initial release"
```

## Submitting to Community Registry

1. Fork [obsidianmd/obsidian-releases](https://github.com/obsidianmd/obsidian-releases)
2. Add entry to `community-plugins.json`:

```json
{
  "id": "my-plugin",
  "name": "My Plugin",
  "author": "Author Name",
  "description": "What the plugin does",
  "repo": "github-username/repo-name"
}
```

3. Create a PR with title: `Add plugin: My Plugin`
4. Complete all checklist items in the PR template
5. Wait for the validation bot:
   - "Ready for review" = passed validation
   - "Validation failed" = fix issues and update release

## Addressing Review Feedback

- Respond to reviewer comments on the PR
- Fix issues in your source code
- Create a new GitHub release with fixes
- Comment on the PR to notify reviewers
- Do **not** create a new PR — update the existing one

Common rejection reasons:
- Using `innerHTML` or other unsafe DOM methods
- Missing error handling for file operations
- Plugin ID conflicts with existing plugins
- Excessive console output in production builds
- Not cleaning up resources in `onunload()`

## Automating Releases

Use GitHub Actions to automate the build and release process:

```yaml
# .github/workflows/release.yml
name: Release Obsidian Plugin

on:
  push:
    tags:
      - "*"

permissions:
  contents: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "22"
      - run: npm install
      - run: npm run build
      - uses: softprops/action-gh-release@v2
        with:
          files: |
            main.js
            manifest.json
            styles.css
```

## Version Bumping

The `version-bump.mjs` script (generated by the `project-setup` skill) keeps `manifest.json` and `versions.json` in sync. It is wired into `package.json` via the `"version"` script:

```json
"scripts": {
  "version": "node version-bump.mjs && git add manifest.json versions.json"
}
```

Usage: `npm version 1.1.0` — this runs the version script automatically, then commits the tag.
